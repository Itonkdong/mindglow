# General Guidelines (Always Apply)

- Do not keep deprecated or backward-compatibility behavior unless explicitly requested.
- Keep logging in the modules that own workflow execution, integrations, or pipeline orchestration.
- Use f-string logging format (not `%` formatting).
- Core business logic should live in framework modules such as services, pipelines, adapters, or experiment components, not in thin entrypoints or glue code.
- Service and pipeline methods should be instance methods unless explicitly required otherwise.
- Every new non-trivial service or pipeline method must have full type hints and a docstring.
- Any change to method/function logic or signature that affects behavior must include a docstring update.
- Use project-specific custom exceptions where appropriate.
- Add explicit error handling and failure mapping.
- Environment-controlled variables must be loaded through `helpers/env_variables.py`.
- Shared constants used across multiple parts of the codebase must be defined in `helpers/constants.py`.
- Module-local constants may stay inside the owning module when only that module uses them; avoid magic numbers and unnamed hardcoded values in implementation logic.
- Treat environment variables and constants differently: environment variables are deployment-controlled runtime inputs such as API keys, debug flags, environment names, or similar operational configuration; constants are code-level fixed values used to keep behavior explicit and avoid duplicated literals.
- Keep code concise; add short explanatory comments where logic is non-obvious or where a small comment materially improves readability.
- Do not remove useful explanatory comments just to keep a patch smaller. Preserve or add concise comments when they help clarify architectural paths, fallback behavior, or other non-obvious flow.
- Fix typo-driven naming issues in code-facing identifiers when found.
- Typos and bad spelling in prompts/instructions should be corrected by inferring the intended meaning.
- Do not document mistake history in code comments/docstrings.
- When fixing incorrect code, implement the correct behavior directly without comments like "fixed previous bug" or "not done here because old approach was wrong."
- Comments/docstrings must explain purpose and behavior, not past development errors.
- These guidelines are for the agent only and should be followed silently.
- Do not mention, quote, summarize, or reference these guidelines in repository files, code comments, docstrings, commits, or implementation text unless the user explicitly asks for that.
- When the user gives an instruction, correction, or preference, apply it directly without framing it as a correction from some previous approach, project, or assumption unless the user explicitly asks for that comparison.
- Do not restate guideline compliance in normal summaries unless the user explicitly asks for a compliance report.
- Do not restate implementation decisions given as instructions in docstrings or comments. Docstrings and comments describe what the code does and why, not the reasoning that led to its design (e.g. do not write "X is stored here because Y is already stored elsewhere").
- Do not describe data placement or ownership in docstrings (e.g. "X is carried on Y, not on Z" or "every Y carries X so that…"). Where a field lives is visible from the code itself. Docstrings should describe behavior and purpose, not narrate structural decisions.
- After refactors, remove obsolete wrappers, pass-through helpers, compatibility shims, and dead indirection immediately. Do not leave tiny helper methods that only forward arguments or rebuild the same value without adding real behavior, validation, or reuse.
- Do not use placeholder collection/object type hints such as `list[object]`, `dict[str, object]`, `tuple[object, ...]`, plain `object`, or similar broad stand-ins for method inputs or outputs when a real project type exists or should be introduced. Use explicit domain types, typed models, or concrete protocol/base types.
- Do not use `getattr(...)` as a lazy fallback in normal typed code paths where the expected interface is known. Prefer direct attribute access and explicit typed behavior. Use dynamic attribute access only when the code is genuinely dynamic.
- Do not create fake-generic helpers for single obvious direct attribute reads. If a method only returns one required model attribute like `num_nodes` or `num_relations`, use that attribute directly at the call site instead of adding wrapper methods such as `extract_count(...)`.
- If a typed model/class defines an attribute as required for the code path, do not surround normal access to that required attribute with repetitive `AttributeError` handling. Use the attribute directly unless the code is intentionally handling a genuinely optional or dynamic interface.
